How I Reduced a Data Pipeline Runtime by Over 60%

The Problem

Every month, a critical data pipeline ran for approximately 2.5 hours before delivering outputs for regulatory reporting. For a pipeline this important — feeding COREP/FINREP submissions — that runtime was a real problem. It compressed the review window, increased the blast radius if something went wrong, and made reruns painful.

The GL (General Ledger) processing step alone was taking ~1.5 hours. That single component was the biggest bottleneck — and the one I focused on first.

Diagnosing the Bottleneck

Before changing anything, I needed to understand why it was slow. I added timing instrumentation across the pipeline and profiled each stage. The picture that emerged was clear:

  • The pipeline was reading large Parquet files (~46M rows) entirely into Pandas DataFrames before doing anything with them
  • This caused a full materialisation of data into memory — a massive, unnecessary data movement step
  • Subsequent SQL transformations in DuckDB were being handed pre-loaded data rather than operating directly on Parquet
  • Orchestration sequencing had implicit idle gaps between stages

The root cause was architectural: the pipeline had been built with Pandas as the central data object, with DuckDB bolted on later. The data was being loaded in full just to be passed to something better suited to handle it natively.

The Solution

The fix was conceptually simple but required careful implementation: remove Pandas from the ingestion path entirely and let DuckDB read Parquet directly via its native file scanning.

DuckDB can query Parquet files with SQL pushdown — meaning predicates, projections, and aggregations are applied at read time without materialising the full dataset. This eliminates the entire "load everything into memory, then filter" pattern that was destroying performance.

Implementation

The key change was replacing this pattern:

Before — Pandas ingestion (the bottleneck)
# Load the entire 46M row Parquet file into memory
df = pd.read_parquet("gl_data.parquet")

# Register as DuckDB table
con.register("gl_data", df)

# Now query it
result = con.execute("""
    SELECT account_code, SUM(amount) as total
    FROM gl_data
    WHERE reporting_date = '2024-01-31'
    GROUP BY account_code
""").df()
After — Direct Parquet streaming into DuckDB
# DuckDB reads Parquet directly with SQL pushdown
# No intermediate DataFrame — predicates applied at read time
result = con.execute("""
    SELECT account_code, SUM(amount) as total
    FROM read_parquet('gl_data.parquet')
    WHERE reporting_date = '2024-01-31'
    GROUP BY account_code
""").df()

This looks like a small change, but the implications are significant. With the read_parquet() approach:

  • DuckDB applies the WHERE filter at Parquet scan time, only reading relevant row groups
  • Column projection means only requested columns are read from disk
  • No Python object is created for the 46M rows — memory pressure drops dramatically
  • The entire operation runs inside DuckDB's vectorised execution engine, which is significantly faster than Pandas for this type of aggregation

Additional Optimisations

Beyond the ingestion change, I made two further improvements:

  • SQL transformation refactoring: Several dbt models had suboptimal join orders and missing indexes on lookup tables. Rewriting these reduced individual query times significantly.
  • Orchestration sequencing: Prefect flow tasks had unnecessary sequential dependencies where parallel execution was safe. Restructuring the DAG reduced idle wait time between stages.

Result

After implementing these changes:

Stage Before After Improvement
GL Processing ~1.5 hours ~30 minutes 67% faster
Total Pipeline ~2.5 hours ~1 hour ~60% faster

Why It Matters

A 60% runtime reduction on a regulatory pipeline isn't just a performance win — it's a risk reduction. When the review window after a pipeline run is measured in hours before a submission deadline, every minute matters.

Faster runtimes mean more time for validation and reconciliation before submission. They also mean reruns are cheaper — so the team is more willing to rerun when something looks off, rather than pushing through with uncertainty.

More broadly, this optimisation illustrates a principle I apply consistently: the best data movement is no data movement. Before adding infrastructure, check whether a tool you already have can handle the problem natively. In this case, DuckDB was already in the stack — it just wasn't being used to its full capability.
