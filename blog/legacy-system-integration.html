How I Integrated a Legacy Acquisition Database into a Modern Analytics Platform

The Real Problem Was Never the Dashboard

When dashboards fail, the visual layer usually gets blamed.

Executives say the numbers cannot be trusted. Teams say reports are slow. Finance exports data into spreadsheets. Operations build side trackers. Everyone assumes the issue is Power BI, Tableau, or the analyst.

In most cases, the dashboard is only exposing a deeper truth: the underlying data model is broken.

I saw this first-hand when our business acquired another specialist accountancy firm. More than a year after acquisition, their operational data still sat outside the main reporting estate. The company had grown through acquisition, but had no repeatable framework for integrating new entities into group reporting.

My task was to change that.

The source system was a legacy SQL-based accountancy backend containing thousands of tables, layered views, inconsistent structures, undocumented logic, and no usable analytical model. The requirement was not to patch a few reports. It was to convert disorder into a production-grade pipeline that aligned with our existing warehouse architecture.

The Objective

I led the analytics delivery across the core implementation:

  • Source system analysis and table discovery
  • Azure Data Factory ingestion pipelines
  • SQL transformation layer design
  • Star schema modelling
  • Deployment into production reporting models
  • Validation with business stakeholders
  • Dashboard integration into the wider reporting suite

The end goal was a clean dimensional model supporting productivity and financial reporting.

That meant producing trusted entities such as:

  • Customers
  • Employees
  • Service Lines
  • Products / Services
  • Operational Activity Facts
  • Revenue and Performance Facts

Once complete, the acquired business could be reported alongside every other entity in the group using the same standards, metrics, and executive dashboards.

The Architecture

The delivery followed a structured warehouse pattern:

End-to-End Flow
Legacy IRIS SQL Backend
        ↓
Azure Data Factory Extraction
        ↓
Raw / Staging Layer
        ↓

        ↓

Simple in diagram form. Considerably less simple in reality.

Technical Challenge #1 — Joins That Multiplied Revenue

One of the most common failures in legacy systems is entity duplication caused by poor relational discipline.

Several operational tables stored customer, job, billing, and service relationships differently. Natural keys were reused inconsistently, and some tables allowed multiple active records where only one should exist.

A straightforward join produced inflated metrics immediately.

Naive Join (Incorrect)
SELECT c.customer_name,
       SUM(i.invoice_value)
FROM invoices i
JOIN customers c
  ON i.customer_ref = c.customer_ref
GROUP BY c.customer_name;

This looked valid. It was not.

Because duplicate customer records existed in the source, invoice rows were being multiplied during joins. Revenue totals became overstated — exactly the kind of silent failure that destroys trust in reporting.

The fix required controlled deduplication using ranking logic and survivorship rules.

Resolved Using Window Functions
WITH ranked_customers AS (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY customer_ref
               ORDER BY last_updated DESC
           ) AS rn
    FROM customers_raw
)

SELECT i.invoice_value,
       c.customer_name
FROM invoices i
JOIN ranked_customers c
  ON i.customer_ref = c.customer_ref
WHERE c.rn = 1;

This established one trusted customer row per business key before facts entered the warehouse model.

Technical Challenge #2 — Inconsistent IDs Across Modules

Another issue was fragmented identity.

The same customer could appear under different references across billing, CRM, time tracking, and historical migrated tables. Names varied slightly, codes changed over time, and some records had no enforced relationship at all.

Without solving identity, there is no reliable star schema.

I introduced conformed surrogate keys in the warehouse and mapped multiple legacy identifiers back to a single analytical entity.

Customer Key Mapping Logic
customer_dim
------------
customer_key
legacy_id
billing_id
crm_id
customer_name_standardised

This allowed facts from multiple source modules to land against one customer dimension, preserving history while enabling consistent reporting.

Technical Challenge #3 — Views Built on Views Built on Views

Legacy systems often accumulate logic through convenience rather than design. In this case, many reporting views referenced other views, which referenced more views. Performance degraded, definitions conflicted, and nobody was certain where certain calculations originated.

Instead of inheriting that complexity, I decomposed the logic into explicit transformation layers:

  • Raw extraction tables
  • Cleaned staging tables
  • Business-rule transformations
  • Dimensional outputs

That made lineage visible, testing practical, and future changes far safer.

The Result

After approximately 3–5 months of phased delivery, the acquired business was fully integrated into the enterprise reporting platform.

Before After
Standalone legacy system Integrated group reporting entity
Manual reconciliation Automated scheduled pipelines
Untrusted joins and fragmented IDs Conformed dimensional model
No acquisition roadmap Reusable integration blueprint

More importantly, this represented roughly 5% of group operations becoming visible through the same executive lens as the rest of the company.

What Leaders Should Understand

Most reporting problems are not reporting problems.

They are modelling problems, ownership problems, and systems integration problems.

A polished dashboard on top of unmanaged source data is decoration. A trusted dashboard built on disciplined modelling becomes infrastructure.

The difference matters.

What I Learned

Technical work becomes valuable when it solves organisational friction.

In this case, the real deliverable was not a dataset or dashboard. It was operational coherence after acquisition — turning an isolated legacy business into part of a measurable, manageable group.

That is where data engineering creates leverage.
